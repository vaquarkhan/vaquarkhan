# Database Reference

## Overview
The Premium Mobility platform persists all domain data in MySQL 8.0. Schema migrations are managed by **Flyway** and the canonical scripts live under `backend/src/main/resources/db/migration`. This directory aggregates:

- `ddl.sql` – consolidated schema definition (kept for quick reference / ad-hoc provisioning).
- `dml.sql` – curated seed dataset aligned with demo UI expectations.
- `README.md` (this file) – descriptive guide to tables, relationships, and seed content.

> **Production note:** Always prefer running Flyway migrations (`mvn -pl backend flyway:migrate`) over executing `ddl.sql` directly. The SQL files remain as documentation and for non-production environments.

## Core Tables
| Table | Purpose | Key Columns |
|-------|---------|-------------|
| `admin_users` | Platform operators with role-based access (`ADMIN`, `CONCIERGE`, `ANALYST`). | `username`, `password` (BCrypt), `role` |
| `customers` | Member profiles, loyalty tiering, preferred language/home airport. | `first_name`, `email`, `loyalty_tier`, `points_balance` |
| `services` | Catalogue of concierge offerings and premium experiences. | `code` (unique), `category`, `base_price`, `premium_only` |
| `chauffeurs` | Chauffeur roster with rating, vehicle, languages. | `name`, `vehicle_type`, `rating` |
| `bookings` | Multi-service booking records (chauffeur, concierge, experiences). | `service_type`, `status`, `pickup_time`, FK → `customers`, `services`, `chauffeurs` |
| `concierge_requests` | Task tickets handled by concierge team. | `request_type`, `priority`, `status`, FK → `customers`, `services` |
| `insurance_products` / `insurance_quotes` | Travel insurance catalogue and issued quotes. | `code`, `max_trip_length_days`, `quote_reference`, FKs → `customers`, `bookings` |
| `esim_packages` / `esim_orders` | Digital SIM inventory and customer orders. | `code`, `region`, `order_number`, FKs → `customers`, `bookings` |
| `partner_offers` | Loyalty-tier gated partner promotions. | `partner_name`, `benefit_summary`, `loyalty_tier_required` |
| `experience_bookings` | High-touch excursions: yachts, helicopter, private jet. | `experience_type`, `operator_name`, FK → `customers`, `services` |
| `loyalty_transactions` | Points accrual/redemption ledger. | `transaction_type`, `points_change`, FK → `customers` |
| `feedback_entries` | Post-service sentiment & ratings. | `rating`, `sentiment_score`, FKs → `customers`, `bookings`, `experience_bookings` |
| `secure_chat_threads` / `secure_chat_messages` | Encrypted concierge conversation threads. | `subject`, `status`, `message`, FK → `customers`, `threads` |
| `translator_sessions` | Offline translator transcripts. | `source_language`, `target_language`, `transcript`, FK → `customers` |
| `biometric_preferences` | Member biometric opt-in settings. | `preference_type`, `enabled`, `metadata`, FK → `customers` |
| `analytics_snapshots` | Hourly KPI snapshots generated by analytics job. | `snapshot_time`, `metric_key`, `metric_value`, `dimension` |
| `ai_interactions` | Audit log for AI endpoints (prompts, responses, latency). | `channel`, `endpoint`, `tokens_used`, `success`, FK → `customers` |

## Relationships Summary
- A `customer` can own multiple `bookings`, `concierge_requests`, `experience_bookings`, etc.
- `services` act as catalogue items linked from `bookings`, `experience_bookings`, and `concierge_requests`.
- `bookings` may reference a `chauffeur`; experience bookings do not.
- Communication modules (`secure_chat_*`, `translator_sessions`) tie back to customers for auditing and personalization.
- Analytics and AI audit tables are append-only and feed dashboards/monitoring.

## Using the SQL Assets
1. **Schema** – Inspect `database/ddl.sql` for whole-of-system DDL. Useful when needing to recreate a local sandbox.
2. **Seed Data** – Execute `database/dml.sql` (after schema is present) to load the curated dataset that the React app expects.
3. **Flyway-first** – In Docker and production pipelines the backend runs Flyway automatically, ensuring migrations remain authoritative.

### Quick Commands
```bash
# Apply migrations locally (requires DB running and proper JDBC props)
mvn -pl backend flyway:migrate

# Load demo data manually (optional; Flyway V1 migration already seeds base rows)
mysql -h localhost -P 33306 -u root -p premium < database/dml.sql
```

## Seed Highlights
- **Users**: `admin/adminpass`, `concierge/conciergepass`, `analyst/analystpass` (dev only; hashed on insert).
- **Concierge Requests**: High, medium priority tasks with due dates and assigned agents.
- **Experiences**: Yacht sunset charter, helicopter cultural tour, private jet summit flight.
- **Communications**: Secure chat threads, translator transcripts capturing real conversation snippets.
- **Analytics**: Snapshots illustrating bookings totals, loyalty movement, AI uptime; AI interactions include latency/tokens for audit.

## Extending the Schema
- Create new Flyway migration files (`V2__feature_name.sql`, `V3__...`) with forward-only changes.
- Update this README with new table summaries and data flow notes when modules are added.
- Keep `ddl.sql` / `dml.sql` synchronized with migrations to support quick local bootstrap scenarios.
